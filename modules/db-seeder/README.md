# clinic-db-seeder

Консольная утилита для наполнения PostgreSQL тестовыми данными ([Faker](https://fakerjs.dev/)). Подключение к БД из Docker ([docker-compose.yml](../../docker-compose.yml) в корне проекта): с хоста используйте `PGHOST=127.0.0.1`, порт `5432`.

## Требования

- Node.js 18+
- Запущенный контейнер Postgres (`docker compose up -d db`)

## Установка

```bash
cd db-seeder
npm install
cp .env.example .env
```

При необходимости задайте `SEED_IMAGES_DIR` — каталог с файлами `.jpg`, `.jpeg`, `.png`, `.webp` для поля `doctors.photo`. По умолчанию берётся `../landing/public/images` относительно папки `db-seeder`.

## Проверка без изменения данных

```bash
npm run verify
```

Выполняет подключение и тестовые вставки в транзакции с `ROLLBACK`.

## Запуск

```bash
npm start
```

В начале можно выбрать **сидер таблиц** или режим **«Приёмы врача на дату»**: массово создать подтверждённые приёмы на выбранный день (предпочтительно будущие слоты; если день уже прошёл или слотов не хватает — времена заполняются по сетке на весь день и при необходимости плотнее) или удалить все приёмы этого врача за дату (сначала удаляются `reviews`, затем `appointments`). Врач задаётся как `doctors.id` или email пользователя из `users`. **Живая очередь** на бэкенде строится только из приёмов с `start_time` в будущем — для прошедшей даты записи в БД будут, в Redis очередь может быть пустой.

Граница календарного дня совпадает с **локальным часовым поясом процесса Node** — для согласованности с бэкендом выставьте ту же TZ, что у JVM (переменная окружения `TZ` / настройки ОС). Живая очередь в Redis пересобирается бэкендом при запросе `GET /api/queue/live/doctor/{id}?date=YYYY-MM-DD` (так же делает кабинет врача). Опционально в `.env`: `API_BASE_URL`, `SEEDER_LOGIN_EMAIL`, `SEEDER_LOGIN_PASSWORD` — после успешного сидера будет выполнен логин и этот GET.

Отдельный вход без полного меню:

```bash
npm run doctor-day
```

В меню сидера таблиц можно выбрать **все таблицы** или отметить нужные, затем указать количество записей (для `roles` выполняется идемпотентная вставка трёх ролей `patient`, `doctor`, `admin` — счётчик не используется).

Порядок вставок соответствует внешним ключам схемы из `sql migration/NEW.txt`.

После всех выбранных вставок автоматически выполняется **дозаполнение `doctors`**: для каждого пользователя с ролью `doctor`, у которого ещё нет строки в таблице `doctors`, добавляется минимальная карточка (иначе в приложении не откроется кабинет врача). Тот же SQL вручную: `sql migration/ensure_doctor_profiles.sql`.

## Примечание по миграции

Если создание схемы падает на индексе `idx_reviews_visible` (колонка `is_visible` отсутствует в `reviews`), исправьте миграцию вручную — seeder схему не меняет.
